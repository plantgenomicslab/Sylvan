## 0. TL;DR

The code is already updated. Two things remain: **swap the container**, and
**delete four stale artifacts** — one of which Snakemake will otherwise never
regenerate, silently dropping an entire evidence track.

```bash
cd /data/gpfs/assoc/pgl/Eli/Btournefortii/Sylvan
```

## 2. What is still wrong, and why a plain re-run would get it wrong

### 2.1 The container is still the pre-fix image

`grep -a mamba_hook` finds **0** hits in the current `singularity/sylvan.sif` and
**9** in `Sylvan_20260708b.sif`. The old image ends its `%environment` with an
unconditional `eval "$(micromamba shell hook …)"`; Apptainer kills any command
substitution in `%environment` that exceeds 5 s, which is what killed 11
`STAR_paired` jobs at container start. `results/logs/miniprot2genewise_.err` still
shows the old image's fingerprint: `eval: 97:43: parameter expansion requires a
literal`.

`singularity/sylvan.sif` is **not tracked by git** (only `singularity/Sylvan.def`
is), so replacing it with a symlink does not dirty the tree and does not involve
Git LFS.

### 2.2 Four stale artifacts — and one of them is invisible to Snakemake

| Artifact | State | Snakemake's view |
|---|---|---|
| `results/TRANSCRIPT/Gmap/gmapExon.gff3` | 0 bytes | job **failed** → incomplete → `--rerun-incomplete` redoes it |
| `results/GETA/homolog/geneRegion_genewise.tmp/` | 804 `.faa` + 804 `.fna`, **0** `.gff` | job **failed** → incomplete → redone |
| `results/.micromamba_hook.sh` | built by the old image | rebuilt only when absent |
| **`results/TRANSCRIPT/Gmap/liftoffExon.fasta`** | **0 bytes** | job **SUCCEEDED** → 🔴 **never redone** |

The last row is the trap, and it is the reason this document exists.

`gmapPrep` ends with:

```bash
bedtools getfasta -fi combined.fasta -fo liftoffExon.fasta -bed liftoffExon.gff3 \
  || touch liftoffExon.fasta
```

On 2026-06-26 `bedtools` was not on `$PATH` inside the `annotation` env —
`results/logs/gmapPrep_.err` records `/usr/bin/bash: line 5: bedtools: command not
found`. The `|| touch` swallowed it, `gmap_build` then ran fine, and the rule
**exited 0**:
`Finished job 0. 1 of 1 steps (100%) done`. Snakemake recorded `gmapPrep` as
complete, and its outputs (11:40:02) are newer than its inputs (11:35:31).

So:

* `--rerun-incomplete` will not redo it — the job did not fail.
* `--rerun-triggers mtime` will not redo it — the outputs are newer.

`gmapExon` *will* re-run (its own output is a failed 0-byte file), it *will* now
pick the correct `gmap` binary — and it will align **an empty FASTA**. The result is
an empty `gmapExon.gff3`, and the entire GMAP liftover-exon evidence track vanishes
from the EVM consensus without a single error message.

**The only way to force `gmapPrep` is to delete its output.** That is what
`preflight_resume.sh --apply` does. (The code fix itself, `d190fcc`, pins the path to
`/opt/envs/alignment/bin/bedtools`, which does exist in the new image — `bedtools` is
declared in the `alignment` env of `Sylvan.def`.)

### 2.3 A bare `./bin/annotate.sh` re-runs Helixer for nothing

Snakemake 7.32.4's default `--rerun-triggers` is
`mtime params input software-env code`. Diffing rule bodies between the run's old
HEAD (`4a6f4cd`) and `v2.1.0`, **10 rule bodies changed**:

```
GETA_output   STAR_single   all                  extractTranscriptsForFilter
gmapExon      gmapPrep      helixer              pickBetterModels
fillingEndsOfGeneModels_ef   fillingEndsOfGeneModels_final
```

`helixer` is in that list and `results/AB_INITIO/Helixer/helixer.gff3` already
exists, so the `code` trigger would re-run Helixer on the GPU and everything
downstream of it. Under `--rerun-triggers mtime` it does not: `helixer.gff3`
(14:05:02) is newer than its input `input/Brassica_tournefortii.fa` (10:39:03).

**`STAR_paired` is *not* in the changed list**, so the 11 BAMs are safe either way.
`STAR_single` is, but this run has no `STAR/single/` outputs.

Hence the recipe: **`--rerun-triggers mtime`, plus delete the artifacts that must be
regenerated.** Deleting an output forces its rule regardless of triggers.

## 4. Procedure

### 4.1 Prepare

```bash
cd /data/gpfs/assoc/pgl/Eli/Btournefortii/Sylvan
git pull --ff-only origin main
bin/preflight_resume.sh . --config input/config_btsm.yml
```

The dry run prints the code check, the container swap it intends to make, a count of
the 11 BAMs it will *not* touch, and the four artifacts it will move. It refuses to
run if `bin/` predates v2.1.0, or if any paired BAM is zero-byte (that is a different
bug — STAR sort RAM — and must be fixed first).

```bash
bin/preflight_resume.sh . --config input/config_btsm.yml --apply
```

Everything moved lands in `results/_preflight_backup_<stamp>/`, including the old
`sylvan.sif` as `sylvan.sif.old`. To roll back, move it all back and `touch` the
restored files.

### 4.2 Resume

```bash
SYLVAN_CONFIG=input/config_btsm.yml ./bin/annotate.sh --rerun-triggers mtime -n
```

Read the dry-run job table before committing. Expect `gmapPrep`, `gmapExon`,
`miniprot2genewise`, one `geneRegion2Genewise` per gene-region group (the crashed
pass had written 804 groups before dying; comparable runs land near 1,000–1,600),
then `merge_geneRegion2Genewise` and `genewiseGFF2GFF3`. Do **not** expect `helixer`,
`STAR_paired`, `STAR_generate`, `RepeatModeler`, `RepeatMasker_*` or `liftoff`. If you
see any of those, stop and work out why before spending GPU hours.

```bash
SYLVAN_CONFIG=input/config_btsm.yml ./bin/annotate.sh --rerun-triggers mtime
```

> `bin/annotate.sh` defaults `SYLVAN_CONFIG` to `config/config_annotate.yml`. This
> run's real config is the untracked `input/config_btsm.yml`, so the variable has to
> be set on every invocation. Untracked files are never touched by `git pull`.

### 4.3 Check afterwards

```bash
ls -l results/TRANSCRIPT/Gmap/liftoffExon.fasta   # must be non-zero now
ls -l results/TRANSCRIPT/Gmap/gmapExon.gff3       # must be non-zero
ls results/GETA/homolog/geneRegion_genewise.tmp/*.gff | wc -l   # one per group
ls -l results/GETA/STAR/paired/*.bam | wc -l      # still 11
```

A zero-byte `liftoffExon.fasta` after the re-run means `bedtools` is still missing
and the container swap did not take.

## 5. Gotchas

Everything in [`HANDOFF_2026-07-08.md` §5](HANDOFF_2026-07-08.md) still applies
(`micromamba activate` in a bare container shell; `TransDecoder --version`; reading a
SIF without singularity; `--fakeroot` builds on the GPU host; never naming a shell
variable `GROUPS`). Additions:

* **`|| touch` turns a missing tool into a silent success.** `gmapPrep` is not the
  only rule that ends this way. When a rule's output is suspiciously empty, check the
  *rule's* log rather than trusting Snakemake's completion state.
* **A `directory()` output is removed recursively before its rule re-runs**
  (`jobs.py:remove_existing_output` → `io.py:_IOFile.remove(remove_non_empty_dir=True)`).
  So `miniprot2Genewise.py`'s bare `os.makedirs(tmp_dir)` is safe under Snakemake, and
  the partial `geneRegion_genewise.tmp/` would have been cleared on re-run anyway.
  `preflight_resume.sh` moves it out regardless, so the state is legible before you start.
* **`bin/rerun_genewise.sh` is a different tool.** It is for runs that already
  *finished* GeneWise with the pre-`5bc3325` worst-locus bug and need it redone before
  EVM. This run never got there. `preflight_resume.sh` detects the difference and says so.
* **The published container directory contains a decoy.** `…/singularity/Sylvan/sylvan.sif`
  is the 2025-05-05 image. Use `Sylvan_20260708b.sif`.
