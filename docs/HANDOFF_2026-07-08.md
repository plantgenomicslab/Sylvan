## 1. What was broken

Five defects, four of which could stop a run outright. They are independent.

### A. `gmapExon` used `gmapl`, which refuses small genomes
`bin/Snakefile_annotate` hardcoded **`gmapl`**, the GMAP build for genomes
≥ 2³² bp (4 Gb). Anything smaller is rejected outright:

```
This program gmapl is designed for large genomes.
For small genomes of less than 2^32 (4 billion) bp, please run gmap instead.
```

*B. tournefortii* is 289,061,629 bp, so the rule could never succeed. A blanket
swap to `gmap` is **not** the fix — the 4.9 Gb *Deschampsia antarctica* genome
genuinely needs `gmapl`. The rule now derives the genome length from column 3 of
the GMAP index's `DB.chromosome` (which it already takes as input) and picks the
matching binary.

### B. `miniprot2genewise` crashed on tab-delimited protein headers
`bin/fasta_utils.py` derived the record ID with `header.split(" ")[0]` — space
only. GWH (Genome Warehouse) protein headers separate their fields with **tabs**:

```
>GWHPCBIT000001<TAB>mRNA=GWHTCBIT000001<TAB>Gene=...<TAB>Position=...
```

so the dictionary key kept embedded tabs, while miniprot writes
`Target=GWHPCBIT000001` — the first token of *any* whitespace. Result:
`KeyError: 'GWHPCBIT000001'` after ~110k mRNAs. In the failing run, 39,063 of
173,874 headers (22.5%) were affected. `readFasta` now takes the first
whitespace-delimited token, per the FASTA convention.

### C. Every container start forked `micromamba shell hook` and could be killed
The image's `%environment` ended with an unconditional
`eval "$(micromamba shell hook …)"`. Apptainer evaluates `%environment` with its
own POSIX interpreter and **kills any command substitution that exceeds 5 s**.
With 11 concurrent 48-thread STAR jobs hammering GPFS, that fork blew past 5 s,
the environment evaluation aborted, and the container never started — the job
died ~8 s in, before STAR emitted a byte:

```
command "/usr/local/bin/micromamba shell hook --shell bash --root-prefix /opt/mamba" was killed after 5s timeout
```

Across 57 job logs the two symptoms are perfectly mutually exclusive: 46 logs
show `eval: 97:43: parameter expansion requires a literal` (the fork succeeded,
then Apptainer's parser choked on the hook's zsh-completion line — harmless),
and 11 show only the 5 s timeout (the substitution was killed, so there was
nothing left to parse).

**The Snakemake-side hook cache cannot prevent this.** `results/.micromamba_hook.sh`
only affects commands run *inside* the container, after `%environment` has
already been evaluated. The fix is at the image level: `%post` now pre-generates
`/opt/mamba/etc/mamba_hook.sh` (with the zsh completion block stripped) and
`%environment` dot-sources it. It also fails the build if that hook is empty,
and tests `-s` rather than `-r` — an empty file is readable, and would have
silently skipped the fallback.

### D. TransDecoder could not load `URI::Escape`
The `genepred` conda env lacked `perl-uri`, so `TransDecoder.LongOrfs` /
`TransDecoder.Predict` died at Perl `BEGIN`-time with `Can't locate URI/Escape.pm`.
These rules sit late in `annotate`, so a run typically reaches them only after
hours of work. Fixed in the def; needs the new image.

### E. GeneWise was handed the *worst* miniprot locus for ~9–15% of proteins
This one silently degrades results rather than crashing.

`bin/miniprot2Genewise.py` keyed regions by the `Target=` attribute and let the
**last** mRNA line win. miniprot emits one mRNA per alignment, ranked best-first
(`Rank=1` is the best hit). Only one region per protein is passed to GeneWise —
so every multi-hit protein was aligned at its worst locus, sometimes on a
different chromosome.

Measured on a real run (*B. tournefortii*, 215,359 mRNA lines / 170,053
distinct targets): **25,896 targets (15.2%) had more than one hit, and 100% of
them kept a non-Rank-1 region.** Example: `ATCG00500.1|PACid:19637947` kept
`scaffold_33` (Rank 5, score 2220) instead of `scaffold_62` (Rank 1, score 2281).
Replaying on a finished *A. thaliana* fleet run: 9,042 of 100,183 proteins (9.0%)
move, e.g. `AL3G47950.t1` from Chr4 to Chr2.

The script now keeps the lowest `Rank`, tie-breaking on the higher alignment
score, without relying on file order.

## 3. How to resume the *B. tournefortii* run

Run dir: `/data/gpfs/assoc/pgl/Eli/Btournefortii/Sylvan` (HEAD `4a6f4cd`, 2026-06-18).

### 3.1 Update the code

Four tracked files were modified in place with copies of newer upstream commits.
They are **byte-identical** to upstream, so discarding them loses nothing:

```bash
cd /data/gpfs/assoc/pgl/Eli/Btournefortii/Sylvan
git checkout -- bin/Snakefile_annotate bin/Snakefile_filter \
                bin/Snakefile_filter_score singularity/Sylvan.def
git pull --ff-only origin main        # 4a6f4cd -> 5bc3325, fast-forward
```

`bin/annotate.sh` carries a **real local edit** (`SYLVAN_CONFIG` defaults to
`input/config_btsm.yml`). Upstream has not touched that file, so the pull keeps
it. Cleaner still: revert it and pass the config on the command line —
`SYLVAN_CONFIG=input/config_btsm.yml ./bin/annotate.sh`.

> If `git` fails with `Error cleaning Git LFS object: … permission denied`,
> the `.git/lfs/tmp` directory is not writable by you. Retry with
> `GIT_LFS_SKIP_SMUDGE=1 git pull --ff-only origin main`.

### 3.2 Point at the new container

The current `singularity/sylvan.sif` is a 10.4 GB **copy** of the May image
(`Sylvan_20260521b.sif`) — it was never built from this repo's def. Replace it
with the published image:

```bash
mv singularity/sylvan.sif singularity/sylvan.sif.old-20260521b   # or delete
ln -sfn /data/gpfs/assoc/pgl/singularity/Sylvan/Sylvan_20260708b.sif \
        singularity/sylvan.sif
sha256sum -c /data/gpfs/assoc/pgl/singularity/Sylvan/Sylvan_20260708b.sif.sha256
```

### 3.3 Clear the artifacts the three failures left behind

```bash
rm -f  results/.micromamba_hook.sh                       # stale: the image changed
rm -rf results/GETA/homolog/geneRegion_genewise.tmp      # 1,608 files, written by the crashed script
rm -f  results/TRANSCRIPT/Gmap/gmapExon.gff3             # 0 bytes, left by the gmapl failure
```

**Keep** `results/GETA/STAR/paired/*.bam` — all 11 are non-zero and valid. The
11 `STAR_paired` errors in the log were *forced* re-runs that died at container
startup; the original BAMs survived.

### 3.4 Re-run

```bash
./bin/annotate.sh          # or: SYLVAN_CONFIG=input/config_btsm.yml ./bin/annotate.sh
```

Snakemake resumes from where it stopped. `gmapExon`, `miniprot2genewise` and the
`STAR_paired` re-runs should now all pass.

## 4. Existing runs that already passed GeneWise

Defect **E** does not crash anything — it quietly degrades the protein evidence.
A run that already produced `results/GETA/homolog/genewise.gff` used the *worst*
locus for every multi-hit protein. Whether to redo that is a scientific call:
the gene models are usable, they are just built from weaker homology evidence.

If you decide to redo it, do so **before the EVM consensus step runs**, otherwise
you throw away that work too. `bin/rerun_genewise.sh` handles it safely:

```bash
bin/rerun_genewise.sh <run-dir>            # dry run: reports groups, jobs, EVM state
bin/rerun_genewise.sh <run-dir> --apply    # backs up, then triggers the re-run
```

It refuses to touch a run whose `bin/` predates the fix, backs up every artifact
the re-run overwrites (per-group GFFs, `genewise.gff`, `genewise.gff3`,
`EVM/genewise*`), and **deletes nothing** — it only `touch`es the checkpoint's
input so Snakemake's `--rerun-triggers mtime` re-runs
`miniprot2genewise → geneRegion2Genewise → merge → genewiseGFF2GFF3`.

Cost: one GeneWise job per ~100 proteins (≈1,000–1,500 per run, 2 CPUs each),
1–4 h wall per run.

If the run's `bin/` predates the fix, refresh it first:

```bash
cp <sylvan-checkout>/bin/miniprot2Genewise.py <sylvan-checkout>/bin/fasta_utils.py <run-dir>/bin/
```

## 5. Gotchas worth knowing

* **`micromamba activate` never works in a bare container shell.**
  `singularity exec <sif> bash -lc 'type -t micromamba'` prints `file`, not
  `function` — Apptainer's `%environment` interpreter cannot export shell
  *functions* to the container process. This is true of every Sylvan image, old
  and new; the pipeline does not depend on it (Snakemake's `shell.prefix` sources
  the hook inside each rule). Test activation as
  `. /opt/mamba/etc/mamba_hook.sh && micromamba activate <env>`.
* **`TransDecoder.LongOrfs --version` does not exist** (`Error, don't understand
  options: --version`, exit 255). Verify with a bare invocation and check the
  usage banner, or `micromamba run -p /opt/envs/genepred perl -MURI::Escape -e 'exit 0'`.
  TransDecoder also exits 255 with no arguments *by design*, so a
  `… | grep -q …` check under `set -o pipefail` reports a bogus failure.
* **Reading a SIF without singularity.** The Pronghorn login/compute nodes have
  no `singularity`, `apptainer` or `unsquashfs`. A SIF embeds its definition file
  in plaintext near the start, so `head -c 1000000 image.sif | grep -a mamba_hook`
  tells you which def an image was built from.
* **Building on the GPU host** (`ssh gpu`, `singularity-ce` 4.4): build under
  `mkdir -m777 /tmp/builddir`. With `--fakeroot` your user maps to a subuid, and
  `/home/pgl` is `drwxr-x---`, so the namespace cannot *traverse* to scratch to
  write the SIF (`Creating SIF file…: permission denied`). `chmod 777` on the
  output directory alone does not help. Do **not** override
  `SINGULARITY_TMPDIR` / `SINGULARITY_CACHEDIR`.
* **Do not name a shell variable `GROUPS`.** It is a bash special array; the
  assignment is ignored, returns non-zero, and kills any script running
  `set -e`.
